# 演讲开场：打破认知惯性

“各位同学大家好。作为软件工程专业的学生，大家平时写代码，不管是做 Web 还是写 App，对‘输入’和‘变量’这两个概念肯定不陌生。在传统的开发视角里，如果用户传了一段文字，我们会把它存进一个 String 变量；如果用户传了一张图片或一段语音，我们把它当作‘附件（Attachment）’，给它一个 URL 存到 OSS 里。

**但是在 AI 时代，尤其是大模型（LLM）向多模态（VLM）跃迁的今天，这种‘附件’和‘字符串拼接’的思维，已经彻底过时了。** 今天我们要聊的，就是 AI 感知系统的代际跃迁，看看现代大模型是如何重构‘输入架构’的。”

---

## 模块一：告别“附件”拼接——原生全模态的架构跃迁

**【讲演逻辑】：** 从传统的流媒体/文件处理流程切入，引出“端到端全模态”的降维打击。

**【配合展示图片】：** 传统多模态模型 vs 全模态模型：视频语音通话实现路径对比

**【讲演话术】：**
“大家先看屏幕上这张对比图。假设我们现在要做一个类似‘科幻电影里的人工智能语音助手’，也就是带视觉和语音的实时通话。

看左边，这是**传统多模态（级联式）**的实现路径。它就像一条臃肿的流水线：用户输入语音，我们得先调一个 ASR（语音识别）服务转成文本；输入视频，得调一个目标检测模型去提取视觉特征。所有的东西最终都要翻译成‘文本’，塞给自然语言大模型去处理。最后，大模型生成了回复文本，我们还得再调一次 TTS（语音合成）把它转回语音。
**工程痛点是什么？** 延迟极高。而且在转换过程中，用户的语气、情绪、停顿，甚至视频里背景的细微光影，全都在‘转成文本’的那一瞬间丢失了。这就叫‘附件处理思维’，各管各的。

现在看右边，这是目前最前沿的**全模态基础模型（端到端）**的实现路径，比如 GPT-4o 或者 Gemini 1.5。
在这套架构里，音频、视频、文本不再是需要互相翻译的独立文件，而是**原生输入**。它们在底层被映射到了一个**统一的多模态特征空间**里。模型不是‘听到了文字’，而是‘听到了音频的波形向量’。这种端到端的处理，不仅让跨模态交互极度流畅、低延迟，更重要的是，AI 第一次真正意义上实现了对多模态的‘无损感知’。”

---

## 模块二：杀死传统 OCR，还是降维打击？—— 空间与语义的融合理解

**【讲演逻辑】：** 顺承模块一的“无损感知”，用具体的场景案例，暴击传统 OCR 的缺陷，展示 VLM 的理解力。

**【配合展示图片】：** 对比 LLM 视觉架构：沙滩场景分析

**【讲演话术】：**
“刚刚讲了架构的跃迁，那落实到具体的业务场景中，差别有多大呢？过去我们要让 AI 看图，最常用的管道就是 OCR（光学字符识别）。但今天我要说，**在很多场景下，单纯的 OCR 管道已经被淘汰了。** 为什么？大家看这张沙滩场景图。

看上面这条**使用外置 OCR 的管道**。OCR 引擎非常尽职尽责，它从图里抠出了所有的英文字符：'BEACH CAFE', 'DANGER', 'WOMAN' 等等。然后把这些词扔给纯文本的 LLM。
LLM 拿到这些词是什么感觉？它就像在做没有上下文的词汇连线题。它知道图里有‘危险’，有‘女人’，但它根本不知道‘危险’是因为旁边有写着强流的警示牌，也感受不到沙滩的阳光明媚。**它看到了字，但丢失了物理空间和视觉上下文。**

现在看下面，**集成多模态 VLM 的特征驱动路径**。
用户的图片通过底层的视觉编码器（比如 ViT），变成高维向量，再通过对齐层直接和文本特征融合。VLM 此时不仅‘认识’牌子上的字，它还理解了：阳光明媚的沙滩上，一位穿着红色比基尼的女性正在休息，远景有咖啡厅，近景有写着‘危险强流’的警示牌。
这就是从‘看见字符’到‘理解场景’的代际跃迁！

**但是，作为未来的软件架构师，我们要有辩证思维。** 传统 OCR 真的死透了吗？并没有。如果你现在的业务是提取银行票据上的 16 位银行卡号，VLM 可能会因为‘幻觉’写错一个数字。所以，现代最先进的架构是**共生路由**：用 VLM 做整体的版面理解和逻辑推理，当遇到需要 100% 准确的局部字符时，VLM 会像调用函数一样，去调用一个极其轻量、精准的 OCR 工具链。”

## 模块三：从“固定提示词”到“上下文工程”—— 动态变量的系统级注入

**【讲演逻辑】：** 把视角从“输入端”拉回“代码端”，讲解现代 AI 变量不再是简单的字符串，而是复杂的流水线模板。

**【配合展示图片】：** 简历筛选 Agent 工作流

**【讲演话术】：**
“理解了 AI 怎么看世界，我们最后来看看在代码层面，我们如何把真实世界的数据喂给它。
很多同学写 Prompt 还是大一新生的思维：用 `f-string` 写一句 `f"请总结这段话：{user_input}"`。这就把动态变量看轻了。在复杂的 AI 应用里，变量的注入是一套完整的**上下文工程（Context Engineering）**。

我们来看这张经典的**简历筛选 Agent 工作流**。
假设你收到了几百份排版各异、不可编辑的 PDF 简历（最左侧）。
在现代工作流里：
第一步，我们利用刚才讲的 VLM（或者 OCR 工具）去解析这张图，把非结构化的画面，变成**结构化的动态变量**，比如提取出 {姓名}、{工作经历}、{技能清单}。
第二步是核心，看中间绿色的框——**填入变量模板**。这里有一个定制化的 Prompt。它不仅包含了你要注入的动态求职者变量，还包含了另一个极其关键的动态变量：【岗位描述 JD】。
这就像是一个复杂的模板引擎。LLM 在做评估时，它不是在泛泛而谈，而是拿着由系统自动抽取的、结构化、高密度的上下文，进行严密的交叉对比，最后输出一个标准化的 JSON 评估报告。

这种把非结构化数据转化为结构化变量，再动态注入到编译好的上下文中的模式，就是目前 AI Agent 运行的基础底座。”

# 结尾过渡与悬念设置

“今天，我们探讨了 AI 是如何通过全模态架构感知世界，以及我们如何在工程上构建强大的动态上下文。

但是，这里有一个巨大的问题留给大家思考：刚才在简历 Agent 里，岗位的 JD 比较短，我们可以直接作为变量传进去。如果我现在有一份 10 万字的《大厂技术面试指南》，或者一家公司 10 年的规章制度，每次提问都要把这 10 万字作为‘动态变量’塞进 Prompt 里吗？那 API 的账单估计会让你破产，且延迟极高。

当‘变量’膨胀成了一座图书馆，我们该怎么办？这就需要引入外部记忆系统和工具调用了。这也是我们下次技术分享的核心主题：**知识库（RAG）与工具调用（MCP）**。感谢大家！”
